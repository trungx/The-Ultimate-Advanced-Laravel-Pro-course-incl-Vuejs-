In this section, we are going to have a thorough introduction to testing laravel applications. The section was recorded using Laravel 5.4, but this does not hinder you from using the latest version of the framework. If you prefer to use Laravel 5.5 for this section, keep in mind the two changes I mention below (these can totally be ignored, but It would help to know that the framework has some new features we can use.):


  1. In Laravel, all exceptions thrown are handled by the App\Exceptions\Handler.php  file. This file takes in the exception, and renders it into a view, and then returns that view to us as the laravel error page. During testing, in versions of Laravel less than 5.5, we manually add throw $exception  in the render  method of the App\Exceptions\Handler.php  file, so as to prevent Laravel from rendering our exceptions into a view, just like it normally does. This helps make testing a lot easier, and we can see what sort of exception we are dealing with. If you are using Laravel 5.5, Laravel provides an inline method to do this, called withoutExceptionHandling() , which you can directly use in your test files.
  2. In Laravel 5.5, the DatabaseMigrations  trait has been replaced with the RefreshDatabase  trait.

If you have no idea what these two points above are about, not to worry. When you see me use throw $exception or DatabaseMigrations in the screencasts, you will understand better, and if you don't, you can always come back to this article to read through again.
